## Changes

### 1. Google social login

- Call `configure_social_auth` with `providers: ["google"]` (keep email enabled).
- Update `src/routes/login.tsx`: add a "Continue with Google" button using `lovable.auth.signInWithOAuth("google", { redirect_uri: window.location.origin })` from `@/integrations/lovable`.
- Keep email/password as secondary option.

### 2. Relax password rules

- Currently `minLength={6}` on signup but Supabase default + HIBP can reject weak passwords. Lower UI `minLength` to 6 (already), and call `configure_auth` with `password_hibp_enabled: false` so common passwords aren't rejected. Keep `auto_confirm_email: false` (user must verify).

### 3. AI chat accessible to everyone + optional save

- `src/routes/support.tsx`: remove the "sign in required" gate. Render the chat UI for guests too.
- Guest mode: chat state lives in component memory (no thread persistence). Show a banner inside the chat UI: *"Sign in to save this conversation"* with a Sign in button.
- Signed-in mode: keep existing threaded sidebar + `/support/$threadId` flow. Add a "Save chat" affordance for newly created guest chats once the user signs in (simple path: after sign-in, the banner becomes "Start a new saved chat" — we don't migrate the in-memory guest messages, to keep scope tight).
- `src/routes/api/chat.ts`: make auth optional — if no bearer token, skip persisting messages to `chat_messages` and just stream the AI response. If authed + `threadId` provided, persist as today.
- `support.$threadId.tsx`: unchanged (authed-only).
- New `src/routes/support.index.tsx` content: render the guest/auth-aware chat directly instead of just a "new chat" splash.

### 4. Granting admin access — instructions (no code change needed)

The `user_roles` table + `has_role()` function already exist. Two options shown to the user in chat:

- **Easiest (one-time bootstrap):** Open Lovable Cloud → Users, find the user's UUID, then run in SQL:
  ```sql
  insert into public.user_roles (user_id, role) values ('<user-uuid>', 'admin');
  ```
- Also add a tiny one-time migration that promotes a specific email to admin — the email is ([newcetf@gmail.com](mailto:newcetf@gmail.com)) and run it.
- then create a admin page that only admins can access, a page that promotes a specific email to admin role to any specific email.

- `lovable.auth.signInWithOAuth` requires the `@/integrations/lovable` module — generated by `configure_social_auth`. Don't hand-write OAuth code.
- `/api/chat` change: read `Authorization` header; if missing → guest path (no DB writes, no `threadId` required). Existing `lookupParcel` tool keeps working for guests (it's a public lookup).
- No DB schema changes needed.
